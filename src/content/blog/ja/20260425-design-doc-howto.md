## 設計書の目次を公開する

[前回の記事](/HomePage/ja/blog/20260424-design-doc-speed/)で、設計書の完成度が開発速度を決めると書きました。では、具体的に何を書けば「完成度の高い設計書」になるのか。

実際に AI 駆動開発で 16 人日 → 2 時間を実現した設計書の構成を公開します。

| # | セクション | 重要度 | 役割 |

## 各セクションの「これだけは書け」

### 技術スタック（★★★）

**バージョンと選定理由**を書く。「React を使う」ではなく「Next.js 15.x (App Router) を使う。SSR/SSG が必要なため」。

```markdown
| レイヤー | 技術 | バージョン | 選定理由 |
|---|---|---|---|
| フロントエンド | Next.js (App Router) | 15.x | SSR/SSG 統合 |
| ORM | Prisma | 7.x | 型安全なクエリ |
```

### データモデル + テーブル定義書（★★★）

**カラム名・型・制約・デフォルト値**まで書く。AI はここが曖昧だと毎回質問してくる。

```markdown
| カラム | 型 | NULL | デフォルト | 説明 |
|---|---|---|---|---|
| id | UUID | NO | gen_random_uuid() | 主キー |
| name | VARCHAR(100) | NO | - | プロジェクト名 |
| status | ENUM | NO | 'planning' | 現在のステータス |
```

### API 設計（★★★）

**パス・メソッド・認証要否・レスポンス型**を一覧にする。

```markdown
| メソッド | パス | 認証 | 説明 |
|---|---|---|---|
| GET | /api/projects | 要 | プロジェクト一覧 |
| POST | /api/projects | 要 | プロジェクト作成 |
| GET | /api/projects/:id | 要 | プロジェクト詳細 |
```

### 権限制御設計（★★★）

**ロール × リソース × 操作のマトリクス**。これがないと AI は「この操作は誰ができますか？」と毎回聞いてくる。

```markdown
| リソース | 操作 | admin | manager | member | viewer |
|---|---|---|---|---|---|
| プロジェクト | 作成 | ✅ | ✅ | ❌ | ❌ |
| プロジェクト | 閲覧 | ✅ | ✅ | ✅ | ✅ |
| タスク | 編集 | ✅ | ✅ | ✅(自分のみ) | ❌ |
```

### 画面遷移設計（★★★）

**ディレクトリ構造**で表現すると、ルーティングと 1:1 対応するため AI が迷わない。

```markdown
app/
├── (auth)/login/          # ログイン画面
├── (dashboard)/
│   ├── projects/          # プロジェクト一覧
│   └── projects/[id]/     # プロジェクト詳細
│       ├── tasks/         # タスク管理
│       └── gantt/         # ガントチャート
```

## 設計書を書く順序

すべてを一度に書こうとすると挫折します。以下の順序で書くのが効率的です。

| 順序 | セクション | 理由 |
|------|-----------|------|
| 1 | 技術スタック | 最初に決めないと後が全部変わる |
| 2 | データモデル + テーブル定義 | データが決まれば API が決まる |
| 3 | API 設計 | データモデルから自動的に導出できる |
| 4 | 権限制御 | API の各操作に対して「誰ができるか」 |
| 5 | アーキテクチャ + 画面遷移 | 全体像を図示 |
| 6 | 残りのセクション | 必要に応じて追加 |

## さいごに

設計書は「書くのが面倒」と感じるかもしれません。しかし、[失敗の科学の記事](/HomePage/ja/blog/20260418-failure-science/)で書いた通り、**最大の失敗は失敗を蓄積しないこと**。設計書は、過去の判断を蓄積し、開発者（人間でも AI でも）が同じ判断を繰り返さなくて済むようにする仕組みです。

★★★ の 7 セクションだけでも書いてみてください。それだけで AI 駆動開発の速度が劇的に変わります。
