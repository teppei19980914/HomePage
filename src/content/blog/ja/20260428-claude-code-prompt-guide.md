## Claude Code、ちゃんと「公式の作法」で使っていますか？

私は AI 駆動開発で個人プロダクトを 3 つ動かしていますが、そのすべてで Claude Code をヘビーに使っています。ただ、最初の頃はずっと感覚的に使っていて、「なんか今日は Claude が賢い」「なんか今日はダメ」と、**Claude の挙動を自分の気分のせいにする** ようなところがありました。

そこから抜け出すきっかけになったのが、**Anthropic の公式ベストプラクティス** をちゃんと読んだことです。

読み終えた瞬間の感想は一つ。

> 「答え、全部ここに書いてあったんだ」

## 何を読んだか

一次ソースは 2 つだけです。

- [Best Practices for Claude Code — Anthropic 公式](https://code.claude.com/docs/en/best-practices)
- [Prompting best practices for Claude 4 — Anthropic 公式](https://platform.claude.com/docs/en/docs/build-with-claude/prompt-engineering/claude-4-best-practices)

英語ですが、**分量はそれほど多くない**。コードと並行して、半日あれば全部目を通せる程度です。それなのに、日本語記事で完全網羅しているものはまだ少ない印象でした。

## 読んだあと、私の使い方はこう変わった

### 1. 「検証手段」を必ず添えるようになった

公式が単独で最も効果が大きい（"the single highest-leverage thing"）と明言しているのがこれ。**テスト・スクショ・期待出力のいずれかを必ずプロンプトに添える** だけで、生成コードの精度が一段上がりました。

### 2. `/clear` を恐れず使うようになった

「同じ指摘を2回以上したら `/clear`」が公式の鉄則。コンテキストが埋まると Claude の性能が劣化する、と公式がはっきり書いています。**セッションを長引かせるほど賢くなる、というのは私の誤解でした**。

### 3. 「いきなり Code」をやめた

Explore → Plan → Code → Commit の 4 段階。**Plan を飛ばしていきなりコードを書かせると、間違った問題に対する正しい解** が出てきます。これを知ってから、複雑なタスクでは必ず Plan Mode を挟むようになりました。

## 信頼度の凡例

本連載では、公式の引用と筆者の解釈を区別するため、以下の凡例を使います。

| 凡例 | 意味 |
|---|---|
| 🟢 公式引用 | 公式ドキュメント原文の直接引用・翻訳 |
| 🟡 公式記述 | 公式ドキュメントに記載された技術・手順 |
| 🔵 筆者解釈 | 公式情報を再整理・優先度付けしたもの |
