# ブログ記事作成スキル

## 記事作成フロー

### Step 0. トレンド分析（毎回必須）

記事の作成・編集を始める前に、最新トレンドを分析して蓄積する。

1. **`analyze-trend` スキルを実行**し、Qiita/Zenn のトレンド上位記事を取得・分析
2. 分析結果が `docs/TrendAnalysis/YYYY-MM-DD_<source>.md` に保存されたことを確認
3. **`docs/TrendAnalysis/TREND_INDEX.md`** の累積パターンを読み込み、以下を把握:
   - 常に有効なパターン（繰り返し確認されたもの）
   - 時期依存パターン（今の季節・イベントで有効なもの）
   - 新興パターン（直近で初めて観測されたもの）
4. 累積パターンと今回の分析結果を踏まえて、以下の Step に進む

### Step 1. テーマ設計

1. ターゲット読者層を明確化（初心者/中級者/上級者）
2. 主要キーワード・副次キーワードを決定
3. **トレンド分析の示唆を参照**し、今取り上げるべきテーマとの接点を特定
4. 競合記事との差別化ポイントを特定
5. 投稿時期とタグ戦略を検討（**TREND_INDEX.md の時期依存パターンを参照**）

### Step 2. タイトル設計（SEO + クリック率）

- **30〜60文字**に収める
- **TREND_INDEX.md の「常に有効なパターン」と「新興パターン」を参照**してタイトル構造を決定
- 基本形: **感情語 + `—` 区切り + 価値提示**の二部構成
- 主要キーワードをタイトル前半に配置
- 「技術」「設計」「戦略」など、チートシートではない深みを示す語を使う
- **今回のトレンド分析で発見した新しいタイトルパターンがあれば積極的に採用**

### Step 3. 記事構成設計

- **冒頭**: 読者の思い込みを揺さぶる問いかけ（「〜と思っていませんか？」）
- **本文**: 段階的構成（初級→上級）で全レベルの読者に発見を提供
- **比較表**: Before/After、従来手法 vs 提案手法を表で可視化
- **実体験**: 自分のプロダクト経験を具体的に引用（一次情報）
- **まとめ**: レベル別アクション表やスキルマップで読後感を明確化

### Step 4. 投稿日と Frontmatter 作成

**投稿日の決定（自動）**:
1. `src/content/blog/ja/` 内の既存ファイル名から最新日付を取得（ファイル名の `YYYYMMDD-` 接頭辞）
2. 最新日付の **翌日** を新記事の `date` に設定（一日一投稿ルール）
3. 例: 最新が `20260418-failure-science.md` → 新記事は `date: 2026-04-19`

```yaml
title: "（Step 2 で設計したタイトル）"
description: "（120〜160文字、主要+副次キーワード含有、記事の価値が伝わる文）"
date: YYYY-MM-DD  ← 上記で決定した日付
tags: ["（具体的なタグ）"]
```

**タグ選定ルール**:
- 汎用すぎるタグは避ける（"AI" → "AI駆動開発"）
- 技術名・固有名を含める
- 季節・イベント系タグを検討（「新人プログラマ応援」等）
- 3〜6個が目安

### Step 5. 記事本文作成

- **ファイル名**: `YYYYMMDD-slug.md`（投稿日の接頭辞 + 英小文字ハイフン区切りのスラッグ）
  - 例: `20260419-new-article.md`
- `src/content/blog/ja/` と `src/content/blog/en/` に同一ファイル名で配置
- URL slug: `YYYYMMDD-英小文字+ハイフン`（スペース・日本語禁止）
- H1 は書かない（frontmatter title が H1 になるため）
- 裸 URL を使わず、説明的なアンカーテキストを使用
- 関連する既存記事への内部リンクを含める

### Step 6. 英語版作成

- `src/content/blog/en/` に同一 slug で英語版を作成
- タイトル・description も英語で SEO 最適化

### Step 7. 既存記事への相互リンク追加

- 新記事に関連する既存記事を特定
- 既存記事にも新記事へのリンクを追加

## 読了時間チェック（記事完成時に必ず実施）

記事完成後、`estimateReadingTime` で読了時間を確認する。**目標: 3〜5分**。

| 結果 | 対応 |
|------|------|
| **3分未満** | 内容の深掘り、または関連記事へのリンクで補完 |
| **3〜5分** | 適正。そのまま進める |
| **5分超** | セクション分割（独立した2記事に）または内容を軽量化 |

**注意**: 無理に水増ししない。短い記事は関連記事リンクで補完する方が読者体験が良い。

記事タイプ別の目安:
- **エッセイ（努力・言葉・MBTI等）**: 3〜4分（一つのメッセージに集中）
- **技術記事（ユメハシ・GitHub等）**: 4〜5分（コード例が必要なため）
- **読書感想**: 3〜4分（「自分がどう変わったか」に絞る）

## SEO チェックリスト（記事完成時に必ず実施）

- [ ] 読了時間: 3〜5分（3分未満は深掘り、5分超は分割を検討）
- [ ] title: 30〜60文字、主要キーワード含有、クリック誘引表現
- [ ] description: 120〜160文字、主要+副次キーワード含有、記事の価値が伝わる文
- [ ] tags: 具体的（汎用すぎないか）、技術名・固有名含有
- [ ] URL slug: 英小文字+ハイフン、スペース・日本語なし
- [ ] H1 重複なし: Markdown 内に `#` 見出しがない
- [ ] 内部リンク: 関連記事への相互リンクが存在する
- [ ] 裸 URL なし: 説明的なアンカーテキストを使用
- [ ] 個人情報なし: 具体的な金額・連絡先・第三者特定情報なし
- [ ] 英語版: en/ に同一 slug で配置済み
- [ ] i18n: 新しい UI テキストがあれば ja.ts → en.ts に追加済み
